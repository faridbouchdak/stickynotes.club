# StickyNotes.club Product Playbook

## Purpose

The Product Playbook translates the Product Constitution into day-to-day
working practices.

Unlike the Constitution, this document evolves as the product matures.
Every guideline should originate from real experience, not theory.

------------------------------------------------------------------------

# Product mindset

Before designing or building anything, ask:

-   What problem is the user trying to solve?
-   Can we make this simpler?
-   Does this fit the existing mental model?
-   Is this consistent with the rest of the product?

If the answer is "no", redesign before building.

------------------------------------------------------------------------

# Writing standards

Every Help Center page should:

-   Answer one user question.
-   Be readable in under two minutes.
-   Use short paragraphs.
-   Avoid unnecessary technical language.
-   End with related guides.

Write about goals, not interface elements.

Prefer:

> Organize your work.

Instead of:

> Use tags and due dates.

------------------------------------------------------------------------

# UX standards

Prefer recognition over explanation.

The interface should make documentation almost unnecessary.

When choosing between two designs:

-   Choose the simpler one.
-   Choose the calmer one.
-   Choose the one that requires less explanation.

------------------------------------------------------------------------

# Navigation

Organise information around user journeys.

Preferred structure:

1.  Get started
2.  Use StickyNotes.club
3.  Manage your account
4.  Community
5.  Need help

Avoid navigation based purely on technical features.

------------------------------------------------------------------------

# New feature checklist

Before adding a feature:

-   Does it help ideas grow?
-   Is it understandable without training?
-   Does it simplify the product?
-   Does it fit Public or Private?
-   Does it require a new Help Center page?
-   Does it introduce unnecessary settings?

------------------------------------------------------------------------

# Documentation workflow

1.  Design the feature.
2.  Write the Help Center page.
3.  Build the feature.
4.  Test with users.
5.  Improve both product and documentation.

Documentation is part of product development, not an afterthought.

------------------------------------------------------------------------

# Tone of voice

Be:

-   calm;
-   practical;
-   optimistic;
-   respectful;
-   concise.

Avoid:

-   marketing hype;
-   unnecessary superlatives;
-   jargon;
-   long explanations.

------------------------------------------------------------------------

# Reflection routine

After every significant feature release:

-   Review the Help Center.
-   Update the Design Notes.
-   Promote proven patterns into the Playbook.
-   Only update the Constitution when a core principle has genuinely
    changed.

------------------------------------------------------------------------

# Golden rule

If a feature cannot be explained simply, it probably is not simple
enough yet.
